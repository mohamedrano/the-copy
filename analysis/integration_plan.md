# مخطط دمج المشاريع الخارجية في "the copy"

يهدف هذا المخطط إلى تتبع تنفيذ الأمر التنفيذي القاضي بدمج التطبيقات الثلاثة المستقلة في تطبيق الإنتاج الرئيسي "the copy"، مع ضمان اكتمال متطلبات النشر وتهيئة خطوط البناء.

## المرحلة 0 — Assemble: تحضير المصادر الخارجية
- [x] إنشاء مجلد `external/` واستيراد المشاريع الفرعية (تم وفق المعطيات الحالية).
- [ ] التحقق من تحديث سكربت PowerShell `scripts/import-external.ps1` بكل المسارات الافتراضية الصحيحة.
- [ ] توثيق استخدام السكربت داخل README (قسم الاستيراد المحلي).

## المرحلة 1 — Grade: تحديث الهوية والعلامة التجارية
- [x] تغيير اسم الحزمة إلى `the-copy` وتحديث العناوين العامة.
- [ ] مراجعة جميع الوثائق (`DOCUMENTATION.md`, `PRODUCTION_*`, إلخ) لضمان اختفاء أي إشارة إلى الاسم القديم.
- [ ] التأكد من أن واجهة المستخدم تعرض "the copy" في العنوان الرئيسي وكل روابط التذييل.

## المرحلة 2 — Mix: ضبط إعدادات البناء للمشاريع الفرعية
- [ ] التأكد من أن كل مشروع Vite خارجي يستخدم قيمة `base` المطابقة لمساره الفرعي:
  - `/drama-analyst/`
  - `/stations/`
  - `/multi-agent-story/`
- [ ] تحديث أي راوتر أمامي (مثل React Router) بخيار `basename` المتوافق.
- [ ] توثيق متطلبات البناء لكل مشروع في ملف `external/README.md` (مقترح).

## المرحلة 3 — Render: تضمين التطبيقات في الواجهة الرئيسية
- [ ] مراجعة مكون `ExternalAppFrame` والتأكد من تغطية حالات التحميل/الأخطاء.
- [ ] وصل صفحات Projects/Templates/Export بالمكون مع تمرير العناوين والمسارات الصحيحة.
- [ ] إضافة اختبارات وحدية بسيطة للتأكد من أن `ExternalAppFrame` يعالج أخطاء `iframe` بشكل صحيح.

## المرحلة 4 — Render (Build Pipeline): نسخ مخرجات المشاريع إلى `public/`
- [ ] تحديث سكربت `scripts/build-external-projects.js` لدعم المسارات الجديدة والتعامل مع السجلات (logs).
- [ ] إضافة مهمة npm `build:external` (موجودة حالياً) إلى مسار `build:prod` لضمان البناء المتكامل.
- [ ] حفظ سجلات البناء لكل مشروع داخل `reports/` لأغراض التتبع (تحسين مستقبلي).

## المرحلة 5 — Export: النشر عبر Docker وCI
- [ ] مراجعة `Dockerfile` متعددة المراحل والتأكد من تثبيت الاعتمادات الضرورية فقط.
- [ ] تحديث `docker-compose.yml` ليتماشى مع المنافذ الحالية وإضافة متغيرات البيئة الاختيارية.
- [ ] التحقق من إعدادات `nginx.conf` لتمكين التخزين المؤقت للأصول ومنع تخزين HTML.
- [ ] ضبط تدفق GitHub Actions (`.github/workflows/build-and-deploy.yml`) للتأكد من نجاح خطوات: التثبيت → الفحوص → البناء → نشر صورة Docker.

## المرحلة 6 — Quality Gates
- [ ] تشغيل `npm run type-check` بعد كل تعديل جوهري.
- [ ] تشغيل `npm run lint` والتأكد من عدم وجود تحذيرات.
- [ ] تشغيل `npm run build:prod` للتأكد من نجاح البناء الشامل.
- [ ] إعداد اختبارات دخان (Playwright) للمسارات الأربعة الرئيسية والتحقق من عدم وجود أخطاء في Console.
- [ ] تجربة حاوية Docker محلياً (`npm run docker:build` ثم `npm run docker:run`) واختبار المسارات وواجهة الصحة `/healthz`.

## المرحلة 7 — المستندات والتقارير
- [ ] تحديث `README.md` لتضمين خطوات تشغيل السكربتات، البناء، والتشغيل داخل Docker.
- [ ] تحديث `SUMMARY_OF_CHANGES.md` في نهاية العمل لتوثيق الملفات المعدلة والاختبارات المجراة.
- [ ] أرشفة أي ملفات قديمة داخل مجلد `archive/` بدلاً من حذفها.

## خطوات المتابعة
1. تنفيذ البنود المتبقية حسب الأولوية أعلاه.
2. بعد كل دفعة تغييرات، الالتزام برسالة Git واضحة (feat/fix/chore/...).
3. تشغيل بوابات الجودة الثلاثية (type-check → lint → build) قبل طلب المراجعة أو الدمج.
4. رفع تقرير مرحلي داخل `reports/` عند اكتمال كل مرحلة رئيسية.

> **ملاحظة:** تم بالفعل نسخ المشاريع الخارجية إلى `external/`، لذا ينصب التركيز الآن على ضمان البناء المتكامل، التضمين داخل الواجهة، وضبط مسارات النشر.
